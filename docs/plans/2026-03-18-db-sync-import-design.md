# 数据库同步与导入能力设计文档

## 背景与目标

当前实现中，数据库同步状态仅保存在运行时内存中。应用重启后，前端会丢失“已同步”状态，用户通常需要再次执行远程同步。虽然数据库文件曾经下载到本地，但现有流程无法在重启后稳定复用该文件。

本设计目标：

1. 避免“每次重启都必须重新同步数据库”。
2. 支持用户设置数据库同步后的本地下载路径（不再固定临时目录）。
3. 在默认保留远程同步能力的前提下，新增本地数据库导入能力作为备用路径。
4. 统一导出 CSV 的数据来源，确保“同步后可导出、导入后也可导出”。

## 范围与约束

- 保留现有远程同步能力，并作为默认模式。
- 新增本地导入模式，不移除现有导出流程。
- 用户界面与提示信息使用简体中文。
- 不改变核心查询/导出业务含义，仅改造数据源管理与路径持久化机制。

## 方案对比与结论

### 方案 A（采用）：双来源 + 持久化数据源配置

- 在查询页新增“数据库来源”区域：远程同步 / 本地导入。
- 远程同步新增“本地落盘路径”配置并持久化。
- 本地导入支持选择 `.db` 文件并校验。
- 导出逻辑统一读取“当前有效数据库路径”。

优点：满足需求完整、改动可控、兼容现有流程。  
缺点：需前后端协同改造。

### 方案 B：前端最小改造

- 仅在前端缓存路径，后端保持现状，靠文件移动拼接流程。

优点：实现快。  
缺点：状态一致性与错误恢复较弱，后续维护成本高。

### 方案 C：后端中心化数据源配置

- 所有数据源状态和路径完全由后端配置管理。

优点：架构最整洁。  
缺点：当前需求下改动偏重，交付成本高。

**最终选择：方案 A。**

## 架构设计

### 前端（Vue + Pinia）

在 `QueryForm` 中新增“数据库来源”区域：

- 来源模式：`remote_sync`（默认）/`local_import`。
- `remote_sync` 显示：远程数据库路径 + 本地落盘路径 + 同步按钮。
- `local_import` 显示：本地数据库文件选择/导入按钮。

在 `query` store 中新增持久化状态：

- `sourceMode`: 当前来源模式。
- `remoteDbPath`: 远程数据库路径。
- `syncTargetPath`: 同步落盘路径。
- `importedDbPath`: 本地导入数据库路径。
- `activeDbPath`: 当前有效数据库路径（导出统一使用）。
- `lastReadyAt`: 最近一次可用状态时间。

### 后端（Rust + Tauri）

扩展命令能力：

1. `sync_database` 支持目标落盘路径参数（例如 `target_path`）。
2. 新增 `validate_local_database` 命令，用于导入后校验文件可读性与关键表存在性。
3. 导出执行前统一校验 `activeDbPath` 可访问性，异常时返回明确中文错误。

## 数据流设计

### 启动恢复流程

1. 前端从本地持久化读取来源模式与路径配置。
2. 若 `activeDbPath` 存在且文件可读，标记为“可直接使用”。
3. 不自动触发远程下载，避免重启重复同步。

### 远程同步流程（默认）

1. 用户填写远程路径 + 本地落盘路径。
2. 点击同步，后端执行远程快照并下载至指定路径。
3. 成功后更新 `activeDbPath = syncTargetPath` 并持久化。
4. 后续导出直接复用该本地数据库。

### 本地导入流程（备用）

1. 用户选择本地 `.db` 文件。
2. 调用 `validate_local_database` 进行校验。
3. 校验通过后更新 `activeDbPath = importedDbPath` 并持久化。
4. 后续导出沿用统一导出流程。

### CSV 导出流程（统一）

1. 导出前检查 `activeDbPath` 是否存在且可读。
2. 根据查询类型调用宽表或需量导出。
3. 输出路径仍通过保存对话框由用户选择。

## 异常与错误处理

1. 参数缺失：未设置远程路径/落盘路径时禁止同步并提示中文错误。
2. 同步失败：若存在上一次可用 `activeDbPath`，继续保留，不覆盖为无效状态。
3. 导入失败：文件不存在、无法打开、关键表缺失时，返回可定位错误信息。
4. 导出拦截：`activeDbPath` 不可用时直接阻断导出并提示“请重新同步或导入数据库”。
5. 路径安全：用户变更路径时不自动删除旧文件，防止误删数据。

## 测试与验收

### 测试场景

1. 首次远程同步成功，导出 CSV 成功。
2. 重启应用后不重新同步，直接导出 CSV 成功。
3. 切换至本地导入，导入成功后导出 CSV 成功。
4. 删除当前活动数据库文件，导出前被正确拦截并提示。
5. 回归验证 SSH 连接、宽表导出、需量导出流程无回退。

### 验收标准（Done）

1. 重启后不再被迫重复同步数据库。
2. 用户可设置同步数据库本地下载路径。
3. 用户可导入本地数据库并完成 CSV 导出。
4. 默认仍为远程同步模式，本地导入为备用模式。
5. 新增提示与错误信息均为中文，且可定位问题。

## 实施建议（高层）

建议按“后端命令扩展 -> store 持久化 -> QueryForm 交互改造 -> 联调验证”顺序推进。  
实现阶段需特别关注：

- 路径跨平台兼容（Windows 路径分隔符与权限）。
- 导入校验覆盖两类导出依赖表（`data_wide`、`demand_results`）。
- 状态持久化与 UI 展示一致性（避免“显示可用但实际不可读”）。
